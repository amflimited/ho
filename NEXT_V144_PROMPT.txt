V144 — PROMPT WORKFLOW REPAIR / GPT TASK PACKETS

Problem observed on branch claude/admin-site-assessment-4Zrig:
- app.php currently treats normal ChatGPT and Custom GPT launches as nearly the same flow.
- cp_gpt_row() builds a ChatGPT URL by appending ?q= to either chatgpt.com or the configured GPT URL. That does not reliably prefill a Custom GPT composer.
- The research tab later recognizes that Custom GPT pages do not honor ?q=, but still uses an href that opens the GPT immediately and only tries to copy the prompt via onclick. On mobile/iOS this is fragile because navigation can happen before clipboard write completes.
- The prompts themselves are huge, stateful, and prone to contamination from prior ChatGPT context. The user is seeing old-region/old-JSON behavior and does not trust the results.

Goal:
Replace the prompt workflow with a reliable task-packet system.

Required architecture:
1. Do not send giant prompts through URLs.
   - Normal ChatGPT deep links may continue to use https://chatgpt.com/?hints=search&q=... only when the encoded URL is below the safe limit.
   - Custom GPT links must NOT append ?q= and must NOT claim “prompt is prefilled.”

2. Add server-side GPT task packets.
   - Add a DB table, for example gpt_tasks:
     id BIGINT AUTO_INCREMENT PRIMARY KEY
     task_uid VARCHAR(64) UNIQUE NOT NULL
     task_type ENUM('sourcing','research','contact','enrichment') NOT NULL
     expected_key VARCHAR(64) NOT NULL
     import_action VARCHAR(64) NOT NULL
     source_run_id BIGINT NULL
     prompt MEDIUMTEXT NOT NULL
     input_fingerprint CHAR(64) NOT NULL
     status ENUM('ready','fetched','imported','failed','expired') DEFAULT 'ready'
     created_at DATETIME NOT NULL
     fetched_at DATETIME NULL
     imported_at DATETIME NULL
     last_error TEXT NULL
   - Create helper functions in ho-model.php:
     ho_create_gpt_task(PDO $pdo, string $type, string $expectedKey, string $importAction, string $prompt, ?int $sourceRunId = null): array
     ho_get_gpt_task(PDO $pdo, string $taskUid): ?array
     ho_mark_gpt_task_fetched(PDO $pdo, string $taskUid): void
     ho_mark_gpt_task_imported(PDO $pdo, string $taskUid): void

3. Add a task fetch endpoint for the Custom GPT Action.
   - New file: gpt-task.php
   - Auth same as gpt-import.php: X-Api-Key, Authorization: Bearer, or ?key=.
   - Accept task_uid via JSON body or query string.
   - Return JSON:
     {
       "ok": true,
       "task_uid": "...",
       "task_type": "research",
       "expected_key": "research_results",
       "import_url": "https://hoosieronline.com/gpt-import.php",
       "prompt": "..."
     }
   - If no task_uid is provided, return the newest ready task so Adam can type/copy only: "Run my next Hoosier Online task."

4. Update gpt-import.php.
   - Accept optional top-level task_uid.
   - If task_uid is present, load the task and validate that the payload includes the task.expected_key.
   - For sourcing tasks, use the task.source_run_id if present instead of requiring the model to preserve run_id perfectly.
   - Mark task imported on success and record last_error on failure.

5. Update app.php buttons.
   - Replace “Open your Custom GPT — prompt is copied, paste & send” with two paths:
     A) Custom GPT Action mode:
        - Create/display task_uid for the current prompt.
        - Button text: "Open GPT task"
        - Copy only this tiny launcher text:
          Run Hoosier Online task TASK_UID. Fetch the task, complete it, and import the results.
        - Then open the custom GPT URL only after clipboard write succeeds.
     B) Normal ChatGPT mode:
        - Keep ?q= deep link only for small prompts.
        - For large prompts, copy prompt and tell Adam to paste or use results.json upload.
   - Implement the open button as JS, not raw href navigation:
     async function hoCopyThenOpen(promptText, url, btn) {
       btn.disabled = true;
       try {
         await navigator.clipboard.writeText(promptText);
         btn.textContent = 'Copied — opening GPT...';
         window.location.href = url;
       } catch (e) {
         btn.textContent = 'Copy failed — tap Copy first';
         btn.disabled = false;
       }
     }
   - Do not advance to the next step until after import succeeds. Current hoAfterGpt() advances after 600ms, before results are imported, which is confusing.

6. Harden all generated prompts.
   Every generated prompt must begin with a reset block like this:

   STATE RESET / CURRENT TASK ONLY
   - Ignore previous conversations, previous JSON, previous businesses, and previous regions.
   - Use ONLY the businesses/region/IDs in this prompt.
   - Do NOT reuse any prior output.
   - Every returned business_id must be one of the allowed IDs listed below.
   - If a fact cannot be verified, use null or an empty string. Do not guess.

   Include a task identity block:
   TASK_UID: <task_uid if available, otherwise omitted>
   EXPECTED_TOP_LEVEL_KEY: research_results | contacts | enrichment_results | candidates
   ALLOWED_IDS: [571,572,...]
   ALLOWED_CITIES: ["South Bend","Mishawaka","Elkhart","Goshen"] where relevant

7. Reduce batch size.
   Current batch sizes are too big for quality:
   - Source: default 12, max 20. Current UI allows up to 50 and defaults 19.
   - Research: max 6–8 because the schema is large.
   - Contact: max 10–12.
   - Enrichment: max 8–10. Current app queues 38, which is a major quality failure.

8. Split research into two stages.
   The current ho_generate_research_prompt asks for nearly everything at once: website audit, Google, Facebook, Instagram, Yelp, Angi, Thumbtack, YouTube, Nextdoor, BBB, reviews, competitors, owner info, age band, service intelligence, etc.
   Replace with:
   - Stage 1: qualification scan — website/contact/google/facebook/basic source links, franchise flag, obvious gaps.
   - Stage 2: enrichment — only for leads that survive stage 1, focused on reviews/competitor/owner/package fields.

9. Source prompt guardrails.
   Add hard rules:
   - City must be exactly one of ALLOWED_CITIES.
   - Do not return Indianapolis, Carmel, Fishers, etc. unless they are in ALLOWED_CITIES.
   - Each candidate must include evidence_url and evidence_type.
   - If evidence_url is not Google Maps, Facebook, or the business website, confidence cannot be high.

10. Import validation.
   - Reject or quarantine any result whose business_id is not in the active task input set.
   - For sourcing, reject candidates whose city is not allowed for the source run.
   - For any URL claim, strip lead-platform pages from website_url as already done, but preserve them in a separate evidence_url if useful.

Acceptance test:
- On iPhone/Safari, tapping the Custom GPT button copies a tiny launcher instruction, then opens the GPT.
- The GPT can fetch the task by task_uid through gpt-task.php and can import through gpt-import.php without Adam pasting giant JSON.
- No button claims a Custom GPT link prefilled a prompt.
- A South Bend / Mishawaka sourcing task cannot produce Indianapolis leads without being rejected at import.
- Research/enrichment prompts no longer ask for 38 businesses or 69 fields at once.
- The app does not advance from Research to Contact/Enrich just because the GPT link was opened; it advances only after import.

Suggested commit title:
V144: replace fragile prompt links with GPT task packets
